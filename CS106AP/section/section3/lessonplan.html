    Lectures #9- More String Loops, Text Files, PyCharm

    We discussed the distinction between strings and integers (specifically, converting between them using the str and int constructors, although we called them functions). Following that, we discussed the loop-count pattern, namely: 

                      
def count_char_occurrences(s, ch):
    count = 0
    for i in range(len(s)):
        c = s[i]
        if c == ch:
            count += 1
    return count

    Next, we talked about reading from a text file. In Python, this looks mostly the same as it does in Java, although with much cleaner syntax. Here's a toy example with all the important ideas: 

                      
def print_text_file(filename):
    with open(filename, 'r') as f: # open a file in read mode ('r') and refer to it as `f`
        for line in f:             # read line up to and including newline character
            print(line, end='')    # each line is just a string, and we specify not to end with a newline when printing
    # file closes automatically at the end of the `with` block, so we can't read any more from it

    Notice that students are now using the print function. This week, we'll be discussing the difference between printing and returning (i.e. side effects vs information flow), and how they are separate. A metaphor I like to use to explain this is the idea of a painter going into a room and painting the wall. Printing is the act of painting, and a return value is the painter coming back and telling you something about what they did inside the room.

    Next, we moved onto running programs from the command line and PyCharm. On the former topic, we didn't discuss it in enormous amounts of depth, but rather just said to type the name of the script, followed by any command-line arguments (which we said was the information the script needed to run), like so: 

                      
$ python3 scriptname.py argument1 argument2

    We haven't discussed sys.argv yet, but will do so this week. That said, if students have questions about this, it's a good idea to review what exactly this line of code is doing. Here's a quick summary you could use:

    When setting up PyCharm on your computer, you installed Python 3.7. This is a program--just like Spotify or Google Chrome--which is able to read .py files and translate them into instructions (essentially 1s and 0s) that your computer can execute. Therefore, when you type python3 scriptname.py, you're telling the python3 program you installed to read the code you wrote in scriptname.py and have your computer run it. In addition, your program might sometimes need additional information, like the filename in the crazy cat program. If that's the case, you specify that information (separated by spaces, if there's more than one bit of information) right after the name of the file. Your program can then access these bits of information.

    Finally, we talked about PyCharm and doctests. This is not the focus of this week's section, but students might have questions about how they work, so here, for reference and because I wrote this without thinking and don't want to delete it:

    • What's PyCharm? How is it different from Parlante.org?

      • Whenever you ran Python Code on Parlante.org, it was actually sent to a server somewhere else, and run on that computer before the results were sent back to you. From now on, you'll be running Python code on your own computer! PyCharm is a piece of software that allows you to edit your Python code, just like Microsoft Word lets you edit text documents. It's called an IDE (short for Integrated Development Environment), because it actually understands Python so can do cool things like identify problems with your code or suggest functions for you to use as you write them.

    • What's a doctest?

      • Whenever we write a function, a crucial decision we need to make is what that function should return given a particular input. A doctest is a way of specifying the expected return value of a particular function on some inputs in the comments for that function. PyCharm makes it super easy to run your doctests to assess that your functions are behaving the way you want them to. You can specify doctests as below: 

                          
def product(num1, num2):
    """
    Returns the product of num1 and num2

    >>> product(6, 7)
    42
    >>> product(8, 10)
    80
    >>> product(0, 100000000)
    0
    """
    # TODO learn math and write this function

      Provided your PyCharm interpreter is set up correctly, you can just right-click on this function and click 'run doctests' and see how many you passed.

argument_exploration.py

import sys # for sys.argv

def main(args):

if __name__ == "__main__":

      And here's what running that program would look like:

Problems

String Parsing

The primary purpose of this problem is to introduce three mechanics, two of which are functional and the last of which is stylistic:

  1. The 'parse pattern' for a string: Given a string with a particular symbol you're looking for and some substring of interest adjacent to that symbol, scan left or right from the index of that symbol.
  2. Looping through a file: Essentially, the loop explained in the file reading section of this handout.
  3. Decomposition: Deciding what parts of your solution you need to decompose into a function and strategically commenting/doctesting them.

The main function

The main goal of this problem is to introduce the mechanics of the main function, as well as to get some practice with working with lists and decomposition.
